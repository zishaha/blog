NanoClaw 技术解析 · 第二篇

NanoClaw 的记忆系统
是怎么设计的？

为什么「把对话存进文件」比向量数据库
更适合个人助手场景

上一篇讲了 NanoClaw 如何通过技能系统分析一份医学报告。文章结尾留了一个悬念：

「下一篇，我会深入讲 NanoClaw 的记忆系统是如何设计的——以及为什么「把对话存进文件」比向量数据库更适合个人助手场景。」

这就是这篇文章。

每次你和 ChatGPT 的新对话开始时，它对你是谁、你上次聊了什么、你的偏好是什么，一无所知。这不是 bug，是设计——大多数 AI 产品把「每次对话独立」当作安全和隐私的保障。

但对于个人助手来说，这是一个根本性的缺陷。

想象一个现实中的秘书：你不需要每次见面都重新介绍自己，不需要重新解释你的工作背景，不需要再说一遍「我上次让你研究的那个方向……」。他记得你，记得你们做过什么，记得哪些结论是已经确认的。

这就是记忆系统要解决的核心问题：让 AI 助手在跨越多次对话之后，依然能建立在已有基础上工作。

◆

一、业界的主流方案：向量数据库 + RAG

大多数有记忆的 AI 助手产品（包括很多企业知识库产品）采用的是这套架构：

对话/文档 → 切片 → Embedding 向量化 → 存入向量数据库
　　　　　　　　　　　　　　　　　　　　　　　↓
用户提问 → 向量检索（余弦相似度）→ 召回 Top-K 片段 → 拼入 Prompt → 生成答案

这套方案有它的合理性：

✅ 向量相似度检索速度快，可以处理海量文档

✅ 语义检索比关键词更灵活，「肺癌手术」能召回「非小细胞肺癌切除术」

✅ 成熟的生态（Chroma、Weaviate、Pinecone……）

但对于个人助手场景，它有一个根本性的问题。

二、向量数据库的三个致命弱点

弱点一：召回是局部的，上下文是全局的

向量检索召回的是片段，不是完整对话。

设想一次深度对话，包含了：

• 报告的目标读者（面向家属，不是医生）

• 重点关注的治疗方向（EGFR 突变靶向治疗）

• 最终用途（发给主治医生参考）

• 文风要求（通俗易懂，避免过度专业术语）

这 4 条信息会被切成独立片段分散在数据库里。下次问「帮我更新上次的报告」，检索可能只召回「EGFR 靶向治疗」那个片段，忘记了「面向家属」「发给主治医生看」这些关键上下文。

片段化是向量 RAG 的原罪。它打碎了叙事的连贯性。

弱点二：Embedding 无法捕捉时序关系

向量相似度是语义的，不是时间的。

「上次我们决定」「之前你说过」「基于上周的结论」——这些时序引用对向量检索来说几乎是透明的。它们的 embedding 向量和当前问题的相似度并不高，很可能被排在后面甚至被截断。

对话的时间线是理解意图的关键，向量数据库天生不擅长这件事。

弱点三：运维成本高，但对个人助手来说并不值

向量数据库需要：维护一个额外的服务、管理 Embedding 模型版本、处理向量老化问题、调优 Top-K 和相似度阈值……

一个个人助手，每天可能只有几十到几百条对话。用向量数据库，是用火炮打蚊子。

三、NanoClaw 的选择：文件系统 + Grep

NanoClaw 的记忆系统设计非常简单，简单到有点出人意料：

/workspace/group/
├── conversations/　　　← 对话归档（按日期命名的 Markdown）
│   ├── 2026-04-02-conversation-1216.md
│   └── 2026-04-06-conversation-1254.md
├── sources/　　　　　　← 生成的文档（研究报告、分析结果）
│   ├── 肺癌早期治疗方案患者家属版_2026.md
│   └── research_20260405_lung_cancer_deep.md
├── CLAUDE.md　　　　　← 持久化的用户偏好 + 工作背景
└── preferences.md　　　← 结构化记忆（可选）

检索机制：不是向量相似度，而是 Claude 直接用 grep、find、Read 工具搜索文件：

# Claude 搜索记忆的方式，跟你在终端用 grep 一模一样
grep -r "肺癌" /workspace/group/conversations/
grep -r "EGFR" /workspace/group/sources/

四、为什么文件系统在这个场景下赢了？

理由一：完整对话，完整上下文

每次对话被完整存档为一个 Markdown 文件。当 Claude 需要回忆某次对话时，读取的是完整的叙事，而不是零散片段——背景、决策过程、最终结论、你当时的措辞偏好，全都在。

理由二：Claude 是最好的「Grep 后处理器」

向量 RAG 的精髓是「让模型从检索结果中提取答案」。但如果检索本身是精确的（Grep），而后处理的模型是 Claude，整个流程的质量反而可能更高：

向量 RAG：模糊检索 + 强模型 = 中等精度
文件 Grep：精确检索 + 强模型 = 高精度（在个人场景下）

Grep 不会有「向量漂移」的问题，不会因为换了 Embedding 模型而让历史记忆失效。

理由三：人类可读，可以手动干预

文件系统里的记忆是纯文本 Markdown，你可以：

• 直接打开文件查看 Claude 记住了什么

• 手动编辑或删除某条记忆

• 在 Obsidian 里用图谱视图看记忆的关联结构

• 用 Git 做版本控制，随时回滚

向量数据库里的 embedding 向量对人类完全不透明。你无法"审查"你的 AI 助手记住了什么。

理由四：零基础设施

NanoClaw 的记忆系统的基础设施需求是：一个文件夹。没有额外服务，没有 API，没有费用，没有故障点。整个记忆系统就是文件系统，跟容器一起启动，跟容器一起备份，Git push 就是同步。

五、记忆的三个层次

NanoClaw 实际上有三个层次的记忆，职责不同，就像 CPU 的 L1/L2/L3 缓存：

🔥 热记忆——当前会话上下文

Claude Code SDK 维护的 session，存放当前进行中对话的完整上下文。这是最新鲜的记忆，Claude 可以直接「看见」而无需检索。每个群组的 session 是隔离的，互不干扰。

SQLite messages 表 → 最近 N 条消息 → 拼入当前 Prompt

🌡 温记忆——归档对话文件

对话结束后，被压缩归档为 conversations/YYYY-MM-DD-conversation-HHMM.md。Claude 不会自动加载所有历史（那样 context 会爆炸），而是在需要时主动检索：

用户：「上次肺癌报告的结论是什么？」
→ grep -r "肺癌" /workspace/group/conversations/ -l
→ 找到文件，读取，然后回答

❄️ 冷记忆——结构化知识文件

sources/ 目录存放已经提炼过的知识文档，比原始对话更精炼，检索效率更高。CLAUDE.md 是元记忆——关于用户偏好、工作背景的结构化描述，每次对话开始时自动加载到系统提示里。

六、CLAUDE.md：最重要的记忆文件

CLAUDE.md 不是「存下来以后可能用到」的档案，而是每次对话都会被加载到系统提示里的活跃记忆。

NanoClaw 架构里有三级 CLAUDE.md：

/workspace/project/groups/global/CLAUDE.md　← 全局记忆（所有群组共享）
/workspace/group/CLAUDE.md　　　　　　　　← 群组记忆（当前群组偏好）
/home/node/.claude/CLAUDE.md　　　　　　← 系统级配置

这相当于每次对话开始时，Claude 都会读一遍「关于你的小抄」：

你的 CLAUDE.md 可以长这样：

用户背景：欧阳哲仁，技术写作者，关注 AI 产品架构和工程实践。

偏好：文章风格技术准确，面向工程师；输出 Markdown；简体中文，专业术语保留英文。

持久化决策：博客使用 Jekyll + GitHub Pages；医学内容一律附加免责声明。

💡 这些信息不需要你每次重复，Claude 每次都知道。这是记忆系统最高效的部分——不需要检索，直接激活。

七、这个设计的边界

当然，文件系统方案也有它的局限，值得坦诚说清楚：

局限 说明
规模限制对话文件增长到几百个时，Grep 性能开始下降。可引入 BM25 本地搜索（如 qmd），无需向量数据库
跨群组记忆每个群组记忆隔离。A 群里学到的，B 群不知道（除非写入全局 CLAUDE.md）
没有语义相似度Grep 只能精确匹配和正则。「上次那个医学相关的东西」这类模糊查询，需要 Claude 自己决定搜什么关键词
依赖主动检索文件系统记忆不会「自动激活」。Claude 需要意识到「我应该去查历史」再主动 grep——这更多是 prompt 设计的问题

八、一个关键观察

个人助手的记忆，本质上是关系记忆
不是知识记忆。

向量数据库擅长回答「关于 X 主题有什么内容」——这是知识型检索，适合企业知识库、文档搜索。

但个人助手的核心价值在于「你了解我」——这是关系型记忆：你知道我的工作背景、我的沟通风格、我们之前做过什么。

关系记忆的特点：

密度低　真正重要的信息并不多，不需要百万量级的向量索引

连贯性强　需要完整叙事，不能碎片化

时序关键　「上次」「之前」「当时我们决定」是核心线索

人格化　「你上次说……」背后有情感和信任

这些特点决定了，对于个人助手场景，一个结构良好的 Markdown 文件，比一个精心调优的向量数据库，更能捕捉「了解你」这件事的本质。

小结

维度 向量数据库 + RAG 文件系统 + Grep
检索方式语义相似度精确 / 正则匹配
上下文完整性片段化完整对话
时序感知弱文件名即时间线
可读性不透明纯文本，人类可读
基础设施需要额外服务零额外依赖
适用规模百万文档级数千对话级
最适场景企业知识库个人助手

NanoClaw 的选择是务实的。它没有因为「向量 RAG 是主流」就跟风采用，而是根据个人助手的实际需求——低规模、高连贯、关系型——选择了最合适的工具。

最流行的方案不一定是最合适的方案。正确的问题是「这个场景需要什么」，而不是「业界都在用什么」。
