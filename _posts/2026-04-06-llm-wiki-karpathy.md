## LLM Wiki

*一种用 LLM 构建个人知识库的模式。*

这是一份思路文档，设计用来直接粘贴给你的 LLM Agent（比如 OpenAI Codex、Claude Code、OpenCode / Pi 等）。它的目标是传递核心思想，具体实现由你和 Agent 协作完成。

---

## 核心思想

大多数人使用 LLM 处理文档的方式是 **RAG**：上传一堆文件，LLM 在查询时检索相关片段，然后生成答案。这能用，但有个根本问题——LLM 每次都在从零开始"重新发现"知识。**没有积累**。如果你的问题需要综合五篇文档的内容，LLM 每次都要重新找到那些碎片、重新拼凑。什么也建立不起来。NotebookLM、ChatGPT 文件上传、以及大多数 RAG 系统都是这种模式。

这里的想法不同。不是在查询时从原始文档检索，而是让 LLM **增量地构建并维护一个持久的 wiki**——一个结构化的、互相链接的 Markdown 文件集合，夹在你和原始资料之间。每当你添加一个新资料，LLM 不只是把它建个索引留着以后用。它会读这篇文章，提取关键信息，然后**把它整合进已有的 wiki**——更新实体页面、修订主题摘要、标注新数据与旧结论的矛盾，在不断演化的综合体上加强或挑战原有观点。知识只编译一次，然后**持续保持更新**，而不是每次查询都重新推导。

这是关键区别：**wiki 是一个持久的、复利增长的产物。** 交叉引用已经在那里了。矛盾已经被标注了。综合已经反映了你读过的所有内容。每添加一个资料、每提一个问题，wiki 都在变得更丰富。

你自己几乎不需要（或很少需要）直接写 wiki —— LLM 负责编写和维护所有内容。你负责找资料、探索、提出正确的问题。LLM 负责其他所有事情——摘要、交叉引用、归档、以及让知识库真正有用的那些枯燥维护工作。我的实际工作方式是：一边开着 LLM Agent，一边开着 Obsidian。LLM 根据我们的对话修改文件，我实时在 Obsidian 里浏览结果——追踪链接、查看图谱视图、阅读更新的页面。**Obsidian 是 IDE，LLM 是程序员，wiki 是代码库。**

这个模式可以适用于很多场景，举几个例子：

- **个人成长**：追踪自己的目标、健康、心理、自我改进——把日记、文章、播客笔记归档进来，逐渐构建出关于自己的结构化图景。
- **深度研究**：在数周或数月内深入某个课题——阅读论文、文章、报告，逐步构建一个有不断演化论点的综合 wiki。
- **读书笔记**：每读完一章就归档，建立人物、主题、情节线索及其关联的页面。读完之后你有了一部丰富的伴侣 wiki。想想 Tolkien Gateway 那样的粉丝 wiki——成千上万个互链的页面，涵盖人物、地点、事件、语言，由社区志愿者花了多年时间建成。现在你自己读书时就能构建出类似的东西，LLM 负责所有的交叉引用和维护。
- **企业/团队**：由 LLM 维护的内部 wiki，从 Slack 对话、会议记录、项目文档、客户电话中持续获取信息。可以加入人工审核环节。wiki 保持更新，因为 LLM 做了那些没有人愿意做的维护工作。
- **竞品分析、尽职调查、旅行规划、课程笔记、兴趣深潜**——任何需要随时间积累知识、希望知识有组织而不是四散各处的场景。

## 架构

共三层：

**原始资料层（Raw Sources）** ——你精心收集的原始文档。文章、论文、图片、数据文件。这些是不可变的——LLM 只读，不修改。这是你的数据源头。

**Wiki 层** ——一个由 LLM 生成的 Markdown 文件目录。摘要、实体页面、概念页面、对比分析、概览、综述。LLM 完全拥有这一层。它创建页面、在新资料到来时更新页面、维护交叉引用、保持所有内容的一致性。你来读；LLM 来写。

**Schema 层** ——一份文档（比如 Claude Code 用的 `CLAUDE.md`，或 Codex 用的 `AGENTS.md`），告诉 LLM wiki 的结构是什么、约定是什么、在摄入新资料、回答问题、维护 wiki 时应该遵循什么工作流程。这是关键配置文件——它让 LLM 成为一个有纪律的 wiki 维护者，而不是一个通用聊天机器人。你和 LLM 随着时间推移共同演化这份文档，摸索出什么对你的领域有效。

---

## 操作

**摄入（Ingest）**：你把一个新资料放进原始集合，告诉 LLM 处理它。一个典型流程：LLM 读取资料，与你讨论关键收获，在 wiki 里写一个摘要页面，更新索引，更新 wiki 中相关的实体和概念页面，在日志里追加一条记录。一个资料可能会触及 10-15 个 wiki 页面。我个人倾向于一次只摄入一个资料并全程参与——读摘要、检查更新、引导 LLM 重点关注什么。但你也可以批量摄入很多资料，监督少一些。这取决于你开发出什么样的工作流，并把它记录在 schema 里供未来的会话使用。

**查询（Query）**：你向 wiki 提问。LLM 搜索相关页面，阅读它们，然后给出有引用的综合答案。答案可以有不同形式，取决于问题——Markdown 页面、对比表格、幻灯片（Marp）、图表（matplotlib）、画布。重要的洞察是：**好的答案可以作为新页面存回 wiki。** 你提出的对比分析、你发现的联系——这些都有价值，不应该消失在聊天记录里。这样，你的探索就和摄入的资料一样，在知识库里不断复利增长。

**检查（Lint）**：定期让 LLM 对 wiki 做健康检查。检查项目包括：页面之间的矛盾、被新资料推翻的过时声明、没有入链的孤儿页面、被提及但缺少独立页面的重要概念、缺失的交叉引用、可以通过网络搜索填补的数据空白。LLM 很擅长建议新的问题方向和需要寻找的新资料。这能让 wiki 随着增长保持健康。

## 索引与日志

两个特殊文件帮助 LLM（也帮你）在 wiki 增长后保持导航能力。它们服务于不同目的：

**index.md** 是内容导向的。它是 wiki 里所有内容的目录——每个页面都有链接、一行摘要，以及可选的元数据（日期、资料来源数量等）。按类别组织（实体、概念、资料来源等）。每次摄入后由 LLM 更新。回答查询时，LLM 先读索引找到相关页面，再深入进去。在中等规模（约 100 个资料、几百个页面）下这个方法效果出奇地好，不需要基于向量嵌入的 RAG 基础设施。

**log.md** 是按时间排列的。它是一份只增不减的操作记录——摄入、查询、检查都记在里面。一个有用的小技巧：如果每条记录都以一个固定前缀开头（比如 `## [2026-04-02] ingest | 文章标题`），日志就变得可以用简单的 unix 工具解析——`grep "^## \[" log.md | tail -5` 就能拿到最后 5 条记录。日志给你提供了 wiki 演化的时间线，也帮助 LLM 了解最近做了什么。

## 可选：命令行工具

到了某个阶段，你可能想构建一些小工具，帮助 LLM 更高效地操作 wiki。对 wiki 页面的搜索引擎是最明显的需求——在小规模下索引文件就够了，但随着 wiki 增长，你需要真正的搜索。**qmd** 是一个不错的选择：它是一个本地 Markdown 文件搜索引擎，支持混合 BM25/向量搜索和 LLM 重排，完全在本地运行。它有 CLI（LLM 可以 shell out 调用）也有 MCP server（LLM 可以作为原生工具使用）。你也可以自己构建更简单的东西——LLM 可以帮你在需要时快速写一个简单的搜索脚本。

## 技巧与经验

- **Obsidian Web Clipper** 是一个浏览器插件，可以把网页文章转成 Markdown。对快速把资料收进原始集合非常有用。
- **本地下载图片**。在 Obsidian 设置 → 文件与链接，把"附件文件夹路径"设置为一个固定目录（比如 `raw/assets/`）。然后在设置 → 快捷键里搜索"Download"找到"下载当前文件的附件"并绑定快捷键。剪藏文章后按快捷键，所有图片都下载到本地。这是可选的，但有用——让 LLM 可以直接查看和引用图片，而不是依赖可能失效的 URL。注意 LLM 不能原生地一次性读取带内联图片的 Markdown——解决方法是先让 LLM 读文本，再单独查看一些或全部引用的图片来获取额外上下文。有点笨拙，但够用。
- **Obsidian 的图谱视图**是看清 wiki 形状的最佳方式——哪些页面相互连接，哪些是枢纽，哪些是孤儿。
- **Marp** 是一种基于 Markdown 的幻灯片格式，Obsidian 有对应插件。可以直接从 wiki 内容生成演示文稿。
- **Dataview** 是一个 Obsidian 插件，可以对页面的 frontmatter 运行查询。如果 LLM 在 wiki 页面里添加 YAML frontmatter（标签、日期、资料来源数量），Dataview 就能生成动态表格和列表。
- **wiki 就是一个 Markdown 文件的 git 仓库。** 版本历史、分支、协作，全都免费得到了。

## 为什么这个方法有效

维护知识库最枯燥的部分不是阅读或思考——而是**记账**。更新交叉引用，让摘要保持更新，在新数据与旧结论矛盾时注明，在几十个页面之间维护一致性。人类放弃维护 wiki，是因为维护成本增长得比价值更快。**LLM 不会感到无聊，不会忘记更新交叉引用，一次可以修改 15 个文件。** wiki 能保持维护，因为维护的成本接近于零。

人类的工作是筛选资料、引导分析、提出好问题、思考这一切意味着什么。LLM 的工作是其他所有事情。

这个思路精神上与 Vannevar Bush 1945 年提出的 **Memex** 相关——一个有关联轨迹连接文档的个人知识库。Bush 的设想比万维网最终成为的样子更接近这里描述的东西：私人的、主动策划的，文档之间的连接与文档本身同样有价值。他没能解决的问题是：谁来做维护？LLM 解决了这个问题。

## 注意

这份文档刻意保持抽象。它描述的是思想，不是具体实现。确切的目录结构、schema 约定、页面格式、工具选择——所有这些都取决于你的领域、你的偏好和你选用的 LLM。上面提到的一切都是可选的、模块化的——取有用的，忽略无用的。比如：你的资料可能全是文字，完全不需要处理图片。你的 wiki 可能小到索引文件就够了，不需要搜索引擎。你可能不关心幻灯片，只想要 Markdown 页面。你可能想要完全不同的输出格式。使用这份文档的正确方式是把它分享给你的 LLM Agent，一起实例化出一个适合你需求的版本。这份文档唯一的工作是传递这个模式。你的 LLM 可以搞定其余的一切。
