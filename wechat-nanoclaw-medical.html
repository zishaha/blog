NanoClaw 技术解析

AI 是怎么读懂
一份医学报告的？

从「帮我分析这份肺癌报告」到交付全过程

几天前，欧阳哲仁让 AI 助手 Andy 分析了一份《早期肺癌治疗方案患者家属版（2026）》医学报告，最终交付了结构完整、引用 NCCN·CSCO·IASLC 权威指南的深度研究文档。

整个过程对用户来说只是「发一条消息、等几分钟、拿到报告」。但在这背后，一套精心设计的系统正在运转。这篇文章拆解它内部的完整机制。

一、从用户视角看：一条消息引发了什么

欧阳哲仁： 帮我深度研究下肺癌早期的治疗方案，需要一篇简体中文报告，专业，准确

Andy 收到这条消息，几分钟后交付了：

  • 一份 Markdown 格式的深度研究报告（专业版，约 8000 字）
  • 一份为患者家属改写的通俗版（含表格、问答、决策流程图）
  • 一份 PDF 版本（排版规范，可直接打印）

用户看到的是「结果」。系统内部发生的，是一条完整的调度链。

二、NanoClaw 是什么？整体架构

NanoClaw 是基于 Claude Code SDK 构建的个人 AI 助手平台。它的特别之处在于：Claude 不只是聊天机器人，而是一个有工具调用能力、有持久记忆、有专业技能包的 Agent。

用户（WhatsApp / Discord / Telegram）
↓
NanoClaw 消息路由层
↓
Claude Code Agent（「Andy」）
↓
技能系统（Skills）
↓
工具链（Web Search / Bash / File I/O / API）
↓
结构化文档（Markdown / PDF / HTML）

其中技能系统是整个流程的核心。

三、技能系统：Claude Code 的「专家插件」

3.1 技能是什么

在 NanoClaw 里，每个技能（Skill）是一个独立的知识包，核心是一个 SKILL.md 文件，它的开头是一段 YAML，定义了技能的触发条件：

---
name: treatment-plans
description: Generate concise medical treatment plans
    in LaTeX/PDF format for all clinical specialties...
allowed-tools: Read Write Edit Bash
---

当 Claude 收到任务，它会扫描技能列表，通过 description 字段语义匹配最相关的技能，然后读取 SKILL.md 作为上下文，按其中的指引执行——等于被「塑形」成了一个领域专家。

3.2 这次分析肺癌报告，激活了哪些技能？

阶段 激活技能 作用
文献检索 literature-review
parallel-web		 搜索 PubMed、arXiv
指南查询 research-lookup NCCN/CSCO/IASLC 指南
PDF 处理 pdf 读取已有 PDF，提取文本
报告撰写 clinical-reports 按临床报告规范组织
患者版改写 treatment-plans SMART 框架，大白话版本

💡 关键点：这不是预设的工作流，而是 Claude 在理解任务后动态规划的调用顺序。每次任务不同，技能组合也会不同。

四、文献检索层：两级引擎协同

parallel-web：全网实时检索

parallel-web 调用 Parallel Web Systems API，这是专为 AI Agent 设计的搜索接口，返回的不是 10 个蓝色链接，而是带引用的综合摘要：

「根据 NCCN 2025 指南（V2.2025），IA 期 NSCLC 首选亚肺叶切除术（基于 JCOG0802 试验，5 年总生存率 94.3%）……[1][2][3]」

research-lookup：学术论文精确查询

对于需要论文级证据的部分，research-lookup 会路由到 Perplexity sonar-pro-search，接入 PubMed、Semantic Scholar、bioRxiv，专攻学术数据库。

两级检索结果合并后，Claude 负责去重、核实、归纳——这就是为什么最终报告里能看到「JCOG0802 + CALGB 140503 双试验验证」这类具体引用，而不是模糊描述。

五、文档生成：从原始内容到结构化报告

professional 版 → 患者家属版：降维改写

treatment-plans 技能的核心是 SMART 目标框架（Specific / Measurable / Achievable / Relevant / Time-bound），它确保每个建议都是「可操作的」：

❌ 「建议手术治疗」

✅ 「IA 期患者：2 周内与胸外科会诊，评估 VATS 亚肺叶切除术适应症，预计住院 3–5 天」

对于医学报告，最终 PDF 优先使用 LaTeX 模板路径——LaTeX 能精确控制医学文档排版规范（字号、行距、表格、脚注格式），比纯 Markdown→PDF 转换质量高一个数量级。

六、记忆系统：知识复利增长

NanoClaw 在文件系统里维护对话归档和生成文档，这意味着：

第一次：「帮我研究肺癌早期治疗方案」→ 检索 + 分析 + 生成（5–8 分钟）

第二次：「把患者版报告发给我」→ 直接读文件（5 秒）

第三次：「基于上次的报告，加入 KRAS 突变的最新进展」→ 增量检索 + 局部更新

这正是 Karpathy 在 LLM Wiki 里描述的「持久知识库」模式：不是每次从零重新分析，而是在已有基础上复利增长。

八、为什么这个架构有效？

专业知识与通用推理分离

Claude 本体擅长推理和语言，但不应该把「如何写临床报告」硬编码在系统提示里。技能系统让这部分可组合、可替换、可独立更新——就像软件的插件架构。

工具调用 vs 幻觉

医学领域容错率极低。实时检索提供带来源的信息，配合 Claude 的推理能力做筛选和综合，比让模型凭训练数据「背诵」医学知识可靠得多。

输出格式的约束力

这些约束不是提示词里的「请做到……」，而是内嵌在技能文档里的结构性约束，模型读了 SKILL.md 之后，等于被「塑形」成了一个临床报告专家。

九、这个模式的边界

  • API 依赖：断网或 key 失效时降级为纯模型能力
  • 技能质量参差：150+ 个技能由不同团队维护，部分指引过于笼统
  • 医学准确性：模型仍然可能误读文献，生成的报告不能替代医生诊断
  • 上下文限制：当报告超过 10 万字，需要分块处理

以下是 Andy 实际交付的《早期肺癌治疗方案患者家属版（2026）》节选，供读者参考 AI 输出质量。

AI 生成 · 基于 NCCN · CSCO · IASLC 2025–2026 指南

早期肺癌治疗方案

患者与家属指南（2026 年版）

🫁 什么是早期肺癌？

早期肺癌 = 癌细胞还"老老实实"待在肺里，没有扩散到其他地方。这个阶段治愈率很高，请充满信心！

早期肺癌的特点：

✅ 肿瘤还在肺内，比较小（通常小于 5 厘米）

✅ 没有扩散到身体其他部位（没有远处转移）

✅ 通常可以通过手术完整切除

🌟 好消息：早期肺癌（IA 期）经过规范治疗，5 年生存率可以达到 85%~97%。越早发现，效果越好。

🔬 重大研究发现：切多少肺？

以前：医生认为必须切掉一整个"肺叶"（约占一侧肺的 1/3~1/2）才算根治

现在：对于小肿瘤（≤2 厘米），只切一个"肺段"，效果与切整叶一样好，甚至生存率更高，肺功能保留更好！（JCOG0802 + CALGB 140503 双试验验证）

手术方式 适合人群 术后肺功能
楔形切除 极小肿瘤（≤1cm） 保留最好
肺段切除 ⭐ 1-2cm 周围型（推荐首选） 明显优于肺叶切除
肺叶切除 肿瘤 >2cm 或靠近中央 标准水平

💊 靶向治疗：EGFR 突变 → 奥希替尼

EGFR 突变是中国肺癌患者中最常见的基因突变，腺癌患者中约 40%~50% 携带。

🔬 ADAURA 试验（2024 年最终数据）

手术后每天口服一粒奥希替尼，连服 3 年：
• 5 年生存率：88% vs 78%（降低死亡风险 51%！）
• 脑转移发生率显著降低
• 目前早期肺癌辅助治疗中最强的生存获益数据

💡 家属提示：如果报告显示"EGFR 突变阳性"，请一定询问医生是否适合使用奥希替尼辅助治疗。国内已纳入医保（2023 年起）。

⚠️ 重要声明：以上医学报告内容由 AI 基于 2025–2026 年公开医学文献生成，仅供参考，不构成医疗建议。具体治疗方案请务必遵从主治医生的专业判断。

小结

「AI 分析医学报告」这件事，表面上是大模型的语言能力，内核是一套工程化的 Agent 架构：

层次 组件 职责
调度层Claude Code Agent任务拆解、技能规划
技能层Skills（SKILL.md）领域知识注入、执行约束
检索层parallel-web + research-lookup实时信息获取、引用溯源
生成层clinical-reports + treatment-plans结构化文档生成
输出层pdf（LaTeX 模板）格式规范化
记忆层文件系统持久化知识积累

这让 AI 不只是「回答问题」，而是在每次对话中构建可复用的知识产出。下一篇，我会深入讲 NanoClaw 的记忆系统设计——以及为什么「把对话存进文件」比向量数据库更适合个人助手场景。
